# Lolly 面向运营方

**一套面向未来、纵深防御的数据防泄漏与溯源策略，伪装成一个创意平台。**

这是一套包裹在你现有工作方式之外的组织免疫系统——让团队每天都需要的日常创意工作发生在你的边界*之内*，而不是从边界泄漏出去。

**这对你有什么好处。** 你将成为那个对既安全*又*受欢迎的方案说“是”的人。你一举堵住了一个数据外泄漏洞，同时清空了设计需求排队——这是少有的、让你更受欢迎而非更招人厌的安全举措。不会再有凌晨三点的电话，因为有人把品牌文件发给了外部承包商，或者把客户数据粘贴进了某个来路不明的网页工具；你要应付的 SaaS 供应商、合同和审计也变少了；而且当有人问起是谁批准了什么时，你都能拿出完整的 git 记录。你可以睡个安稳觉。

Lolly 作为一款创意工具当之无愧：它清空了设计排队，把可直接投产的成果交到每个人手中。但它之所以能够如此大范围地*安全*分发，原因在于架构本身。没有任何内容会被上传，一切都可复现，并且每一次导出都可以携带一份加密记录，说明它的来源。本页讲述的正是这套安全性与推广落地的故事。

> **先说实话。** Lolly 的安全特性*在设计上*就很过硬，其加密与文件解析引擎目前正在接受 SUSE 严格的基础设施加固，为企业级规模做准备——这方面我们确实很擅长。下文提到的签名封印、设备端签名与加密都是真实且站得住脚的；在这项加固完成之前，请把它们当作纵深防御的一层，而不要当成在合同要求独立保证的场合可以直接采信的认证控制项。我们更愿意事先把这一点说清楚。

## 战略优势

日常创意工作通常的完成方式，本身就是一个风险暴露面：文件通过邮件发给外部设计承包商，品牌素材被上传到十几个 SaaS 编辑器，客户数据被粘贴进陌生人的网页工具里，只为“快速做张图”。这些做法无一例外，都是数据在脱离你的掌控。

Lolly 把这个局面反转了过来。那些*引发*外泄的工作——语录卡片、本地化横幅、活动徽章、打码截图——现在都在员工自己的设备上完成，套用你的品牌，全程没有服务器介入。你不是在高风险的流程上叠加一层管控，而是用一个从一开始就不存在外泄路径的流程，取代了那个高风险的流程。

- **配置权在你手中。** 引擎与各个 shell 均为开源（MPL-2.0）。你可以叠加自己的身份认证、遥测或 CA；可以自行托管，也可以不托管；功能与成本完全由你掌控，一切都有 git 记录可查，不会被锁死在某个 SaaS 数据库里。
- **治理是数据，不是仪表盘。** 工具目录（catalog）是唯一的可信来源，以 Git 仓库的形式管理——pull request 审核*就是*内容审核，员工能接触到的每一个模板，你都拥有完整的审计记录并可即时回滚。参见[采用与治理](/info/adoption-governance.html)。
- **护栏是结构性的。** 品牌约束被硬编码进模板本身，而不是作为可以被人忽视的指导方针发布出来。错误的产出不是被“不鼓励”，而是根本无法产生。

## 在扩大内容产出的同时清空需求队列

Lolly 的目标之一是**设计需求分流**：让常规请求永远不需要送到设计师手上，因为需要素材的人自己就能在几分钟内正确地做出来。每一张被分流掉的工单，既是效率上的胜利，也意味着少一份文件在人与人之间流转。

Lolly 的设计初衷是贴合你的组织实际运作方式——部署它并没有唯一正确的方式：

- **分发部署，而非集中托管。** 通过你现有的 MDM（Intune、Jamf、Munki 等）把 Lolly 推送到各设备。它以桌面/移动应用或离线 PWA 的形式在本地运行——可在任何防火墙之后、任何隔离网络环境中使用，无需维护服务器，更新节奏完全由 IT 掌控。
- **仅集中托管。** 在你的网络内部（或 VPN 之后）运行一个实例；用户通过浏览器访问，无需安装任何东西。发布一次工具，所有人立即可用；搭配你的身份提供方（IdP）实现访问控制。
- **混合模式。** 离线外勤用本地应用，借用设备用始终最新的浏览器版本——两者指向同一个工具库。

完整的部署模式与管理流程说明，参见[部署](/info/deployment.html)与[配置](/info/configuration.html)。

以上这些都是设备端本地转换：你的文件或数据进去，干净的字节出来，而且**没有任何服务器可以上传**。它们是刻意反其道而行之的产物——与那种善意员工原本会去使用的、典型的“把文件上传到陌生网站清洗一下”的工具截然相反。



## 确定性与可复现性

每一个工具的输入都可以表示为 URL 参数，相同的输入总是产生相同的文件。这对运营方来说有两个直接影响：

- **URL 本身就是产物。** 提交这个链接，按需重新生成素材即可——不必把二进制文件提交进 Git，也不必在聊天记录里追查“最新版本”是哪一个。素材与工具的 ID 是永久性的约定，因此今天生成的链接，日后依然能够正确解析。
- **CLI 与 GUI 走的是同一条渲染路径**，因此构建流水线与应用程序永远不会产生偏差。可以在构建阶段以可复现的方式生成 OG 图片、社交卡片和数据可视化图表。

## 溯源与 Content Credentials

导出的文件可以携带 **Content Credentials**——一份签过名的 [C2PA](https://c2pa.org) 清单，与文件字节的哈希值绑定。这是**可察觉篡改，而非防篡改**：它并不能阻止任何人修改文件，但之后任何改动都会破坏这枚封印，支持 C2PA 的验证工具会将其检出。这是一个诚实且实用的特性——你可以在离线状态下，通过密码学手段*察觉*到内容是否被改动过。

- **默认开启，设备端完成。** 签名密钥在设备上生成，不可导出（连 Lolly 自己都读不到），签名过程在本地完成——只有可选的身份*注册*环节才会涉及网络。
- **信任分级。** 未注册身份的导出文件在结构上是有效的，但属于匿名签名（`untrusted`）。注册一个**已验证身份**（由 Lolly CA 签发的短期证书，与邮箱绑定）后，锁定 Lolly 根证书的验证工具会报告 `trusted` 及签名者的邮箱。可信时间戳机构与第三方验证工具的绿色认证（C2PA 合规）已列入路线图，尚未上线——各信任等级均如实标注，文件绝不会显示虚假的绿色认证。
- **凭证有效期**由运营方/用户在签名时自行决定：7 / 30 / 90 / 365 天，默认 30 天。
- **验证在设备端完成。** 把任意文件拖放到 `/valid`（或运行 `lolly validate <file>`），即可离线获得一份报告，说明它是否确实由 Lolly 制作、且此后未被修改。参见 [Content Credentials 身份](/info/content-credentials-identity.html)。

> **已知缺口，直说无妨：** Lolly 的验证工具目前还不能完整读取其他生产方生成的 C2PA **v2** 版声明清单；而且 WebM 是以 Matroska 附件形式携带清单的（WebM 目前尚无标准化的 C2PA 映射方案），因此第三方工具能验证 Lolly 的 MP4，却验证不了它的 WebM。

## 加密与密码保护

对于必须加锁传输的文件，一切都在设备端完成：

- **PDF 打开密码**——*标准*级别是 40 位 RC4，起威慑作用（可在任何地方打开，可随链接传播）；*强*级别是 **AES-256**（PDF 2.0），在导出时输入，且绝不会出现在链接中。
- **锁定下载**——一个 ZIP、一个 Projects 文件夹，或一次批量运行的结果，都可以整体加锁：*标准*为 ZipCrypto（较弱但通用性强），*强*为 **AES-256**（WinZip AE-2）。纵深防御体现在：强级别压缩包内的任何 PDF，也会*同时*被单独加上 AES-256 锁，解压后依然保持锁定。
- **密码保护的分享链接**——整个链接状态都在一个由 PBKDF2 派生出的密钥下进行 AES-256 加密；传播的只有密文，密码从不出现在链接中，解密在接收方的浏览器中完成。

## 隔离网络就绪

Lolly 在设计上可以做到**渲染时完全不依赖网络**。Web shell 是离线优先的 PWA（service worker）；字体和 WASM 都存储在设备本地；工具状态通过 host bridge 在本地持久化，绝不使用 `localStorage`。任何需要联网的工具，都必须通过其清单中声明的、经过**白名单限制**的 `host.net` 能力才能访问网络——无法（或不愿）满足该能力的 shell 会以空实现代替。因此，一套完全隔离网络的安装，其渲染、导出、加密与凭证验证全过程都无需向任何外部“回传”数据。

## 在依赖它之前，你必须了解的事

运营方理应了解这些附加说明，而不只是听到宣传语：

- **面向企业级规模的加固。** 正如开头所说——加密与解析模块目前正在接受 SUSE 面向企业级规模的严格基础设施加固；它们在设计上是过硬的，但在合同要求独立保证的场合，请将其视为纵深防御的一环。
- **工具钩子*不是*安全沙箱。** 工具可选的 `hooks.js` 运行时会注入 host bridge，但在浏览器 shell 中，它是在页面的作用域内执行的，*可以*访问 `window`/`document`/`fetch`。请像对待任何你要运行的代码一样对待工具代码——审查它。这正是“目录即 Git 审核”模式如此重要的原因，也是为什么在 Worker 隔离机制上线之前，不应运行不受信任的第三方工具。
- **C2PA 是可察觉篡改，而非防篡改**，上文提到的 v2 读取缺口与 WebM 缺口都是真实存在的。
- **不同加密等级效果不同。** *标准*级别的锁只起威慑作用；只有*强*级别（AES-256）才是真正的保护，且强级别加密的文件并非所有旧版阅读器都能打开。
