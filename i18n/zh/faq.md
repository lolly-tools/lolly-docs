# 常见问题

在 `/info` 着陆页手风琴组件中展示的常见问题。

**如何维护:** 下面每个 `##` 标题都是一个问题;紧随其后
(直到下一个 `##` 为止)的内容就是它的答案。答案使用与本站
其他部分相同的轻量级 markdown——段落之间用空行分隔。可以在此
添加、删除或重新排序问题,然后重新运行 `npm run build:info`(或
`npm run dev:web`)。第一个 `##` 之前的所有内容(本标题及这些说明)

## 在 /profile 页面选择加入后会发生什么?

初次使用 Lolly 时,你在任何地方输入的所有内容都是完全私密的,直到你主动决定通过媒体或分享链接(如果联网)把这些信息公开出去为止。

选中该选项后,我们会把你的部分个人资料信息作为溯源信息嵌入到资源和包中,以标识你为来源。

Lolly 会生成大量内容。我们采取严格的数据最小化方针来降低风险。

### 什么是功能开关(feature flags)?

功能开关用于开启或关闭 Lolly 的部分功能。通常这些开关由管理员掌控——而在 Lolly 中,由你自己掌控。

## 如何获取移动版或桌面版应用?

任何人都可以分发自己的应用,这些应用所包含的工具和配置会因目标受众不同而有很大差异。因此并不存在唯一的一款应用,除非是你自己制作的,或者是相关的人提供给你的。

## 为什么叫 "Lolly Tools" 这个名字?

**Lolly** 因为自由是甜的。
**Tools**(工具)在不被使用时处于闲置状态。不会窥探你,不会运行秘密程序,
把它投入使用,一切听你号令、由你行动、由你定规则。

**Lolly** 是澳大利亚、新西兰、英国方言里对"糖果"或"甜食"的说法。就像糖果一样,工具对需要它们的人来说也非常"美味"。

我们也在为通过这种方式省下的时间和账单而窃笑。

## 采用 Lolly 会遇到哪些阻力?

Lolly 可以嵌入到任何你已经在生成文件的场景中——CLI 与 App 用的是同一个引擎,
因此凌晨两点跑的流水线不会与某人在浏览器里预览到的结果产生偏差。采用过程中的
阻力很少是技术性的,而是组织性的。可以预期以下几点:

**工具和品牌目录都必须有人编写。** Lolly 是一个平台,而不是一套现成的模板包。
对于*受管理的部署*而言,需要有人来定义共享资源目录(logo、调色板、字体,均以
永久 ID 的形式),并为每种输出类型编写清单(manifest)和模板(template)。不过,
个人用户不必等待这一步——在开放版 App 中,任何人从第一天起就可以把自己的文件
导入目录,并在 Layout Studio 中构建工具。

**依托 git 的治理方式是可选的——而且对非工程师来说很陌生。** 如果你运行的是
一个*共享的、受控的*目录,"PR 审查*就是*审核把关"这一点,对工程师来说很优雅,
但对大多数品牌和市场团队来说却很陌生。如果掌握品牌决策权的人不在 git 里工作,
你就需要一套能够连接双方的工作流——否则 IT 部门就会不知不觉地变成战略性的
设计伙伴,以及更广义上的机构把关人。在许多长期运行的生产环境中,这种做法反而
更受青睐。不想采用这种方式的团队,大可以跳过它。

**它的定位刻意收窄——请这样理解它。** Lolly 不适用于定制化或"英雄级"内容。
它*确实*是你的个人 DAM——由你的设计系统、工具与目录驱动并增强——它也*确实*
拥有一个开放画布(Layout Studio),但即便在那里,颜色、字体和资源仍然遵循当前
生效的设计全局变量,因此自由排布始终被限定在系统之内。若拿去和 Figma 或 Canva
相比,它会显得受限。但若按其本来的定位来评判——可运营化、可重复、大规模的
资源生成——则无出其右。错误的定位认知是最常见的挫折来源。

**生产端的变更管理。** 现有流程即便产出内容不符合品牌规范,今天依然能用。
把它们重新指向引擎,意味着要重新测试、重新学习,而"我们已经能做出文件了"
就会成为不迁移的借口。可以先转换一个高曝光度、生产级质量的输出,并把迁移
前后的效果并排展示出来。

Lolly 会把一切都提升上去。


## 实用工具(utilities)与工具(tools)有什么区别?

**基础答案 →** 实用工具并不总是需要渲染,因此可以采用不同的用户体验(UX)。 

**真正的答案 →** 实用工具之所以能托管在 Lolly Tools 内部,是为了再加一层"便利层"防御,以打消数据外泄的动机。 

为什么?因为众所周知,每天都有人把**自己手头已有的机密内容**交给某个
随机网站,只为完成一个微小的机械操作:

- "**压缩这份 PDF**" → 把合同 / 工资单 / 董事会文件上传给未知的第三方。
- "**把 HEIC 转成 JPG**" → 把带有 GPS EXIF 信息的个人照片上传给靠广告盈利的托管方
- "**裁剪 / 调整这张图片大小**" → 上传产品截图或尚未发布的素材
- "**格式化这段 JSON**" / "解码这个 JWT" → 把 API 响应、令牌、密钥粘贴进某个格式化工具
- "**合并这些 PDF**" → 上传**两份原本绝不该出现在同一台服务器上的文件**

这些网站以及数量庞大的同类克隆站点**默认并不可信**——保留期限未知、
司法辖区未知、次级处理方未知,再加上广告/联盟营销的商业模式,让它们完全有
动机留存你交给它们的东西。操作本身微不足道;**代价却是你的内容。** 

我们要靠出色的便利性和服务,来赢得这场关于治理的战争。 

## Lolly 能编辑并渲染我的 Figma、Penpot、Illustrator 或 InDesign 文件吗?

可以。打开 **Layout Studio** 并点击 **导入设计**:它支持原生 Figma **.fig** 文件(通过 Save local copy 保存的本地副本)、Penpot 的 **.penpot** 导出文件、Illustrator 的 **.ai** 或 **.pdf**、InDesign 的 **.idml**(File → Export → InDesign Markup),或**任意 SVG**(这是最宽的入口——几乎所有设计软件都能导出 SVG)。一切解析都完全在你的设备本地完成,无需账号,也无需插件。

图层会以可编辑方框的形式出现在开放画布上:文字保持可重新输入、形状保持为形状、图片会加入你的本地资源库,字体和颜色则遵循品牌全局设置。保存后,该版式就会成为一个可复用、可通过 URL 访问的模板,任何使用 Lolly 的人都能重新填充内容——你还可以混入实时工具(比如二维码、图表),它们会在加载时重新渲染。此后,它就会像 Lolly 中的其他任何内容一样渲染——SVG、PDF、PNG 等等,并且可以通过其 URL 复现。参见[导入设计](/info/design-import.html)。

## Lolly 能否为已有的 PowerPoint 演示文稿更换品牌?

可以——有两种方式,且都在你的设备本地完成。**Rebrand a Deck** 实用工具接受一个 `.pptx` 文件,把其中的主题、硬编码的颜色和字体替换为你的品牌样式,而图表、SmartArt 和动画则原样保留——你得到的仍是一个 `.pptx` 文件。或者,在 **Deck Builder** 中打开该演示文稿(Load → 拖入文件),把每一页幻灯片当作自由对象逐页编辑,内容已自动贴合品牌规范,并可导出为 PPTX、PDF 或视频。若改为把 `.pptx` 拖放到上传区域,则会把你选中的幻灯片以 SVG 资源的形式归入你的资源库。参见[导入设计 → 演示文稿与文档](/info/design-import.html#decks-and-documents)。

## 8 月 29 日会发生什么?

SUSE 品牌的工具将从项目中移除,由用户自行定义的全新通用示例工具接替。

SUSE 将运营自己的 Lolly 实例,以保护其商标。

## SUSE 保留了多少私有内容?(也就是,什么时候会"过河拆桥")

SUSE 的商标和品牌工具仅用于演示,期限到 8 月 29 日为止。你可以在 [lolly.ART](https://lolly.art) 找到一个无品牌版本的 Lolly 实例。

SUSE 是一家企业级开源基础设施公司,在平台领域拥有三十多年的领先地位。其产品包括企业级 Linux、云原生、边缘计算和 AI 基础设施解决方案。

从 SUSE 的角度来看,这关乎在主权与安全方面言行一致。截至今天,SUSE 将 Lolly 产品化的可能性几乎为零。

完全公开说明:SUSE *确实*在构建内部工具,以将 Lolly 整合进其 IT 系统——这属于 SUSE 内部的搭建工作,与公开开发还是私有开发无关。

说到公开的一面,Lolly 的目标是通过 [Open Build Service](https://openbuildservice.org/) 构建,并由 [SUSE Application Collection](https://apps.rancher.io/applications) 交付安全的供应链制品。

我们会尽可能多地公开构建——只是你不会长期看到 SUSE 品牌的工具,也不会看到 SUSE 的内部员工团队与商业流程,因为这些都与 Lolly 无关。
