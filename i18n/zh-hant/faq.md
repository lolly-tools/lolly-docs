# 常見問題

在 `/info` 首頁的手風琴選單中顯示的常見問題。

**維護方式：** 以下每個 `##` 標題都是一個問題，標題以下的內容（直到下一個 `##` 為止）就是該問題的答案。答案使用與網站其他部分相同的輕量級 markdown 語法──以空白行分隔段落。可以在此新增、刪除或調整問題順序，然後重新執行 `npm run build:info`（或 `npm run dev:web`）。第一個 `##` 之前的所有內容（本標題與這些說明）在建置時會被忽略。

## 在 /profile 頁面選擇加入（opt-in）後會發生什麼事？

第一次使用 Lolly 時，你在任何地方輸入的內容都是完全私密的，除非你刻意透過媒體或分享連結（若為線上）將該資訊公開。

勾選加入之後，我們會將部分個人資料以來源標記（provenance）的形式嵌入到資產與套件中，以標示你為內容來源。

Lolly 會產出大量內容，因此我們採取嚴格的資料最小化原則以降低風險。

### 什麼是功能旗標（feature flags）？

功能旗標可以開啟或關閉 Lolly 的部分功能。這類設定通常由管理員控管──但在 Lolly，由你自己掌控。

## 如何取得行動版或桌面版應用程式？

任何人都可以發布自己的應用程式，這些應用程式所包含的工具與設定，會依目標受眾的不同而有很大差異。因此並不存在單一的官方應用程式──除非是你自己建置的，或是相關人士提供給你的。

## 為什麼取名為「Lolly Tools」？

**Lolly** 因為自由是甜的。
**Tools（工具）** 不使用時處於休止狀態，不會窺探你，也不會偷偷執行程式，
一切聽你指揮：你的指令、你的動作、你的條件。

**Lolly** 是澳洲、紐西蘭、英國用語，意指「糖果」或「甜食」。就像糖果一樣，工具對有需要的人來說也非常「美味」。

我們也樂於見到這個做法為我們省下的時間與費用。

## 導入 Lolly 可能會遇到哪些障礙？

Lolly 可以無縫嵌入你現有產生檔案的任何流程──CLI 使用的引擎與 App 完全相同，因此凌晨兩點自動執行的流程結果，絕不會與有人在瀏覽器中預覽的結果有落差。導入時的阻力很少來自技術面，多半來自組織面。以下是常見的幾種：

**工具與品牌目錄需要有人建置。** Lolly 是一個平台，不是一套現成的範本包。若是*治理式導入（governed rollout）*，就需要有人定義共用的資產目錄（標誌、色票、字型皆以永久 ID 呈現），並為每種輸出類型撰寫 manifest 與範本。不過個人使用者不必等待這一步──在開放版應用程式中，任何人從第一天起就能把自己的檔案匯入目錄，並在 Layout Studio 中建置工具。

**以 git 治理是可選項──而且對非工程師來說並不熟悉。** 如果你採用的是*共用、受控管*的目錄，「PR 審查*就是*把關」這個概念，對工程師而言相當優雅，但對多數品牌與行銷團隊來說卻很陌生。如果掌管品牌決策的人不熟悉 git，你會需要一套能夠銜接雙方的工作流程──否則 IT 部門就會悄悄變成事實上的策略設計夥伴，以及更廣泛的機構把關者。而這正是許多長期營運的正式環境所偏好的做法。不想要這麼做的團隊，大可略過這一步。

**它刻意做得很窄──也該用這個角度來理解它。** Lolly 不是用來做客製化或形象主打（hero）內容的工具。它*就是*你的個人 DAM──由你的設計系統、工具與目錄所填充、強化──它*也*有一個開放畫布（Layout Studio），但即使在那裡，色彩、字體與素材仍會遵循目前生效的設計全域設定，因此自由編排始終保持在系統範圍之內。拿它跟 Figma 或 Canva 相比，會顯得很受限；但若以它真正的定位來評判──也就是流程化、可重複、大規模的素材產出──那就沒有對手了。用錯框架來理解它，是最常見的挫敗原因。

**產出端的變革管理。** 現有流程即使產出偏離品牌，今天依然能運作。要把它們改接到引擎上，代表需要重新測試、重新學習，而「我們本來就做得出檔案」就會成為不遷移的藉口。建議先從一個能見度高、正式品質的產出開始轉換，並把轉換前後的結果並列展示。

Lolly 會把一切都提升一個層次。

## 公用程式（utilities）與工具（tools）有什麼不同？

**簡答 →** 公用程式不一定需要即時渲染，因此可以採用不同的使用者體驗（UX）。

**真正的答案 →** 之所以能在 Lolly Tools 內建置公用程式，是為了再增加一層「便利性防線」，藉此降低資料外洩的誘因。

為什麼？因為眾所皆知，每天都有人把**自己手上原本就有的機密內容**，交給某個隨機的網站，只為了執行一個小小的機械式操作：

- 「**壓縮這份 PDF**」→ 把合約、薪資單或董事會簡報上傳給不明對象。
- 「**把 HEIC 轉成 JPG**」→ 把帶有 GPS EXIF 資訊的個人照片，上傳到靠廣告營利的空間主機。
- 「**裁切／調整這張圖片大小**」→ 上傳產品截圖或尚未發布的素材。
- 「**格式化這段 JSON**」／「解碼這個 JWT」→ 把 API 回應、token、機密資料貼到格式化工具裡。
- 「**合併這些 PDF**」→ 把**兩份原本絕不該共用同一台伺服器的文件**一起上傳。

這些網站，以及數量龐大、彼此複製的長尾同類網站，**預設就不值得信任**：資料保留時間不明、司法管轄不明、次要處理者不明，再加上廣告／聯盟行銷的商業模式，讓它們有十足的誘因留下你交出去的東西。操作本身微不足道，**代價卻是你的內容。**

我們靠卓越的便利性與服務，來贏得這場治理之戰。

## Lolly 能編輯並算繪我的 Figma、Penpot、Illustrator 或 InDesign 檔案嗎？

可以。開啟 **Layout Studio**，點選 **匯入設計稿**：它接受 Figma 原生格式 **.fig**（另存本機副本）、Penpot 匯出的 **.penpot**、Illustrator 的 **.ai** 或 **.pdf**、InDesign 的 **.idml**（檔案 → 匯出 → InDesign Markup），或**任何 SVG**（這是最寬廣的一扇門──幾乎所有設計軟體都能匯出 SVG）。所有解析都完全在你的裝置上進行，不需要帳號，也不需要外掛。

圖層會以可編輯方塊的形式出現在開放畫布上：文字仍可重新輸入、形狀仍是形狀、圖片會加入你裝置上的素材庫，而字體與色彩則遵循品牌全域設定。儲存之後，這個版面就會變成一個可重複使用、具備網址（URL）的範本，任何使用 Lolly 的人都能重新填入內容──你甚至可以混搭即時工具（例如 QR code、圖表），它們會在載入時重新算繪。之後，它就會像 Lolly 裡的其他任何內容一樣輸出──SVG、PDF、PNG 等等，並可從其網址重現。參見[匯入設計稿](/info/design-import.html)。

## Lolly 能為既有的 PowerPoint 簡報重新套用品牌嗎？

可以──有兩種方式，且都在你的裝置上完成。**改編簡報品牌（Rebrand a Deck）** 這個公用程式會讀取一份 `.pptx`，把主題、寫死的色彩與字型替換成你的品牌設定，而圖表、SmartArt 與動畫則維持原樣、不受影響──最後你會拿回一份 `.pptx`。你也可以在 **Deck Builder** 中開啟該簡報（載入 → 拖放檔案），逐張投影片以自由物件的方式編輯，且已自動套用品牌設定，並可匯出為 PPTX、PDF 或影片。若你改為把 `.pptx` 拖放到上傳區域，則會把你選擇的投影片存為 SVG 素材，加入你的素材庫。參見[匯入設計稿 → 簡報與文件](/info/design-import.html#decks-and-documents)。

## 8 月 29 日會發生什麼事？

SUSE 品牌工具會離開這個專案，改由使用者自訂的全新通用範例工具接手。

SUSE 會營運自己的 Lolly，以保護其商標。

## SUSE 保留了多少私有內容？（也就是所謂的「rug-pull」會在何時發生）

SUSE 的商標與品牌工具僅供展示之用，效期至 8 月 29 日為止。你可以在 [lolly.ART](https://lolly.art) 找到未使用品牌包裝的 Lolly。

SUSE 是一家企業開源基礎架構公司，在平台領域擁有超過三十年的領導地位。其產品涵蓋企業級 Linux、雲原生（Cloud Native）、邊緣運算（Edge）以及 AI 基礎架構解決方案。

從 SUSE 的角度來看，這是在自主權與安全性上「說到做到」。就目前而言，SUSE 將 Lolly 產品化的可能性幾乎等於零。

誠實揭露：SUSE *確實*正在建置內部工具，把 Lolly 整合進其 IT 系統──但這屬於 SUSE 內部的設置安排，與公開版或私有版的開發無關。

說到公開的一面，Lolly 的目標是透過 [Open Build Service](https://openbuildservice.org/) 建置，並由 [SUSE Application Collection](https://apps.rancher.io/applications) 提供具備安全供應鏈保障的產出物。

我們會盡可能公開建置──只是你不會再長期看到 SUSE 品牌工具，也不會看到 SUSE 內部人力與商業流程，因為這些都與 Lolly 無關。
