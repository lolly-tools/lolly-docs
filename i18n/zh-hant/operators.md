# Lolly 給維運人員

### 一套經得起未來考驗、多層防護、防資料外洩、可追溯來源的策略——只不過它剛好是一個創意生產平台

這是包覆在你既有工作流程外的組織免疫系統——讓團隊每天需要的例行創意工作，發生在你的防護邊界*之內*，而不是外洩到邊界之外。

**這對你有什麼好處。** 你可以成為那個對一件既安全*又*受歡迎的事說「好」的人。你一舉堵住外洩漏洞，同時刪掉設計需求排隊清單——這是少見會讓你更受歡迎、而不是更不受歡迎的資安勝利。不會再有凌晨三點的緊急電話，因為有人把品牌檔案寄給外部承包商，或把客戶資料貼到某個不知名的網路工具裡；要處理的 SaaS 供應商、合約與稽核也變少了；還有一份完整的 git 紀錄，當有人問「這是誰核准的」時，你隨時能拿出來。你可以安心睡覺了。

Lolly 之所以能作為創意工具佔有一席之地：它刪掉了設計排隊清單，並讓每個人手上都能產出可用於正式發布的成果。但它之所以能*安全*地廣泛發放，原因在於架構本身。沒有任何東西會上傳，一切都可重現，而且每一次匯出都能附上加密紀錄，證明它的來源。這個頁面談的就是安全性與推行的完整故事。

> **目前的狀態。** Lolly 的安全特性在設計上就很強大，而它的加密與檔案解析引擎正在接受 SUSE 企業級的基礎架構強化。以下提到的封印、裝置端簽署與加密，現在都是真實且站得住腳的，並正逐步邁向獨立認證——因此在合約要求認證級保證之處，請在該流程完成期間，將它們部署為多層防護的一環。

## 策略優勢

例行創意工作平常的完成方式，本身就是一個風險暴露面：檔案被寄給外部設計承包商、品牌素材被上傳到一堆 SaaS 編輯器、客戶資料被貼到陌生人的網路工具裡只為了「隨手做張圖」。這些每一項，都是資料離開你掌控範圍的時刻。

Lolly 把這個順序反過來。那些*導致*外洩的工作——語錄卡片、在地化橫幅、活動識別證、去識別化的截圖——現在都在一個工具上完成，這個工具運行於員工自己的裝置上，套用你的品牌，過程中完全沒有伺服器參與。你不是在高風險的工作流程上疊加一層控管，而是直接把這個高風險的工作流程，換成一個從一開始就沒有外洩路徑的流程。

- **設定權在你手上。** 引擎與各個殼層（shell）都是開源的（MPL-2.0）。你可以疊加自己的驗證、遙測或憑證機構（CA）；可以自行架設，也可以不架設；功能與成本的完整控制權都在你手上，一切以 git 追蹤紀錄，不會被鎖在某個 SaaS 資料庫裡。
- **治理可以是資料，而非儀表板。** 當你想要那種掌控時，可以把工具目錄（catalog）當作 Git 儲存庫來管理——pull request 審查就成了品牌審核，你的員工能接觸到的每一個範本，都有完整的稽核軌跡與即時回溯能力。這是一種選項，而非義務：只想做東西的團隊，可以在版面工作室（Layout Studio）中自行編寫工具、把自己的檔案匯入目錄，全程在應用程式內完成，完全不必碰 git。詳見[採用與治理](/info/adoption-governance.html)。
- **護欄是結構性的。** 品牌限制是直接寫死在範本裡，而不是發布成一份可以被人忽略的指引文件。錯誤的產出不是被「勸阻」——而是根本無法產生。

## 刪掉需求排隊清單，同時讓內容量大增。

Lolly 的目標之一是**分流設計需求**：讓例行需求完全不需要送到設計師手上，因為有需求的人自己就能在幾分鐘內正確做出來。每一張被分流掉的需求單，既是生產力上的勝利，也代表少一個檔案在人與人之間流轉。

Lolly 的設計是為了配合你的組織實際的運作方式——部署方式沒有單一「正確答案」：

- **部署到裝置，而非集中託管服務。** 透過你既有的 MDM（Intune、Jamf、Munki……）把 Lolly 推送到裝置上。它以桌面／行動應用程式或離線 PWA 的形式在本機執行——不管在任何防火牆之後、任何實體隔離（air-gapped）環境中都能運作，沒有伺服器需要維護，更新節奏完全由 IT 掌控。
- **僅集中託管。** 在你的內網中（或 VPN 之後）執行單一個實例；使用者透過瀏覽器存取，不需安裝任何東西。工具只要發布一次，所有人立刻都能使用；搭配你的身分識別提供者（IdP）做存取控制。
- **混合模式。** 離線外勤工作用本機應用程式，借用的機器則用隨時保持最新的瀏覽器版本——兩者指向同一個工具庫。

完整的部署模式與管理操作說明，請見[部署](/info/deployment.html)與[設定](/info/configuration.html)。

## 確定性與可重現性

每個工具的輸入都能以 URL 參數表示，相同的輸入永遠會產生相同的檔案。這對維運人員來說，有兩個實際意義：

- **URL 本身就是產出物。** 把連結提交（commit）進版控，需要時再依需求重新產生素材——不用把二進位檔案放進 Git，也不用在聊天紀錄裡追著找「最新版本」是哪個。素材與工具的 ID 是永久不變的約定，所以今天產生的連結，之後依然能正確解析出結果。
- **CLI 走的是與 GUI 完全相同的渲染路徑**，所以建置流程（build pipeline）與應用程式不會出現落差。可以在建置時就以可重現的方式產生 OG 圖片、社群卡片與資料視覺化圖表。

## 來源追溯與 Content Credentials

匯出的檔案可以附帶 **Content Credentials**——這是一份已簽署的 [C2PA](https://c2pa.org) 資訊清單（manifest），與檔案位元組的雜湊值（hash）綁定在一起。之後對檔案的任何更動都會破壞這個封印，因此支援 C2PA 的驗證工具能**在離線狀態下，以加密方式偵測出竄改**。這份憑證是*可偵測竄改（tamper-evident）*的：它會標示出竄改，而非阻止竄改——這正是讓完全離線驗證得以成立的關鍵。

- **預設開啟，裝置端運作。** 簽署金鑰在裝置上產生，無法被匯出讀取（連 Lolly 自己都讀不到），簽署過程完全在本機進行——只有選用的身分*註冊*程序才會連上網路。
- **信任層級。** 未註冊身分的匯出檔案，結構上仍然有效，但屬於匿名簽署（`untrusted`）。註冊**已驗證身分**（由 Lolly CA 核發、綁定電子郵件的短效憑證）後，信任 Lolly 根憑證的驗證工具就會回報 `trusted`，並顯示簽署者的電子郵件。可信任時間戳記機構與第三方驗證工具的綠燈認證（C2PA 相容性）都還在規劃中。每個層級都是明確的，檔案只會宣稱它能證明的信任等級。
- **憑證效期**由維運人員／使用者在簽署當下自行決定：7、30、90 或 365 天，預設為 30 天。
- **驗證在裝置端進行。** 把任何檔案拖放到 `/valid`（或執行 `lolly validate <file>`），就能取得一份離線報告，判斷這個檔案是否真的由 Lolly 產生、且自產生後未曾更動過。詳見 [Content Credentials Identity](/info/content-credentials-identity.html)。

> **互通性說明。** Lolly 目前已能在離線狀態下驗證自己的憑證，以及許多第三方的憑證。目前有兩項互通性工作正在進行中：完整讀取其他產生者所產出的 C2PA **v2** 版資訊清單，以及 WebM——它目前還沒有標準化的 C2PA 對應方式，所以 Lolly 會把資訊清單以 Matroska part 的形式附加上去（第三方工具能直接驗證 Lolly 產生的 MP4；WebM 則要等標準底定後跟進）。

## 加密與密碼保護

對於必須以加密狀態傳遞的檔案，所有處理都在裝置端完成：

- **PDF 開啟密碼**——*標準（Standard）*等級是 40 位元 RC4，屬於嚇阻性質（可以在任何地方開啟，也可以透過連結傳遞）；*強（Strong）*等級是 **AES-256**（PDF 2.0），在匯出時輸入密碼，且密碼絕不會出現在連結中。
- **加密下載**——ZIP 檔、Projects 資料夾，或一次批次執行的結果，都可以整批加密鎖定：*標準*等級用 ZipCrypto（強度弱，但相容性最廣），*強*等級用 **AES-256**（WinZip AE-2）。多層防護的做法是：Strong 等級 zip 檔裡的任何 PDF，也會*額外*個別以 AES-256 鎖定，所以解壓縮之後依然保持鎖定狀態。
- **需要密碼的分享連結**——整個連結狀態都會用 PBKDF2 衍生出的金鑰以 AES-256 加密；傳遞的只有密文，密碼絕不會出現在連結裡，解密則是在接收者的瀏覽器中進行。

## 支援實體隔離環境

實體隔離是一種**一等公民的部署方式**，而非特殊模式——Lolly 開箱即用，渲染時完全不需要網路。網頁殼層（web shell）是離線優先（offline-first）的 PWA（透過 service worker 實現）；字型與 WASM 都儲存在裝置本機；工具狀態透過主機橋接（host bridge）保存在本機，絕不使用 `localStorage`。任何需要連上網路的工具，都只能透過一個**白名單制**、且必須在其資訊清單（manifest）中宣告的 `host.net` 能力來連線——無法（或不願意）提供這項能力的殼層，會把它做成一個空的替代實作（stub）。透過你的 MDM 把殼層推送到裝置上，或在你的內網中執行單一個實例，一套完全實體隔離（air-gapped）的安裝環境，依然能夠渲染、匯出、加密，並驗證憑證，完全不需要對外連線回報任何資訊。

## 值得知道的事

在你推行之前，有幾件事值得先弄清楚：

- **強化作業進行中。** 加密與解析器目前正在接受 SUSE 企業級規模的強化（見上文）——在設計上今日已經很強；在合約要求認證級保證之處，請把它當作多層防護的一環來部署。
- **工具掛鉤（hooks）*不是*安全沙箱。** 工具選用的 `hooks.js` 雖然執行時會注入主機橋接（host bridge），但在瀏覽器殼層中，它是在頁面的執行環境（realm）裡運作，*可以*存取 `window`／`document`／`fetch`。請用對待任何你要執行的程式碼的方式來對待工具程式碼——審查它。這正是為什麼一個運行共用目錄的組織，可以透過 Git 審查來把關；無論採取哪種方式，在 Worker 隔離機制推出之前，都只執行你已審查過的工具。
- **Content Credentials 是可偵測竄改，而非防止竄改。** 它們偵測更動，而非阻止更動——詳見上方的互通性說明。
- **兩種加密層級。** *標準*等級的鎖定是快速、通用的嚇阻手段；*強*等級（AES-256）才是完整保護——任何敏感內容都應選用 Strong，但要注意它需要較新的閱讀器。
